You'll create an organization, choose a **monitored process** template, connect or seed an **evidence source**, run a first scan, review findings, and generate an audit-ready snapshot. No credit card is required.

<Callout type="info">
A workspace is the tenant-local home for process configuration and runtime evidence. The configuration remains file-backed under `.kiket/`, so changes are versioned and reviewable, but the app guides the setup without asking first-time users to understand git.
</Callout>

## Walkthrough

<Steps>

<Step>

### Create your account

Visit [app.kiket.dev](https://app.kiket.dev/register) and sign up with email or a Google / Microsoft / GitHub account. You'll land in an empty organization — its name is taken from your email domain, you can rename it later.

</Step>

<Step>

### Pick the first outcome

In onboarding, choose the outcome you want to monitor first. Kiket shows what the template creates before you commit:

- process states and case types
- expected evidence
- likely scanner findings
- evidence sources to connect
- report output

For the fastest path, choose **Explore sample data** or a template with a built-in evidence path such as production change approval.

</Step>

<Step>

### Create your workspace

Create the organization and workspace that will own the monitored process, cases, evidence, findings, remediation, and reports. You can rename both later.

The setup flow writes the selected process configuration into `.kiket/`, creates the workspace record, and opens the product around the first-value checklist.

</Step>

<Step>

### Connect an evidence source

Open **Settings → Evidence Sources** and connect the source that can prove the process happened. Typical first sources are:

- GitHub for pull requests, reviews, commits, and deployment evidence
- Jira or Linear for work item status and ownership
- Slack or Teams for approval and incident context
- webhook/API ingestion for internal systems

Each source produces raw events, normalized operational events, evidence records, and scanner context.

</Step>

<Step>

### Run the first scan

After one case and at least one evidence record exist, run or wait for the first scanner pass. The scanner compares the process model with observed reality and creates findings such as:

- missing approval evidence
- evidence source disconnected
- stale access review
- SLA risk or breach

</Step>

<Step>

### Resolve one finding

Open **Findings** or **Remediation**, attach missing evidence or assign an owner, and close the loop. Kiket keeps the finding explanation, source event, evidence, remediation action, and audit history together.

### Generate the audit snapshot

Open **Audit Room** and generate a report for the monitored process. The report pulls from the same graph: process model, cases, evidence, findings, remediation, scanner runs, and optional integrity proofs.

</Step>

<Step>

</Steps>
